# NIST 800-171 hardening (defensibility upgrades for a CUI enclave)

Baseline 800-171 is met by Google's defaults (immutable Cloud Audit Logs `_Required` sink for 400 days, FedRAMP-authorized US services, IAM-based access control). The items below are **hardening upgrades** for a CUI enclave handoff — *none* is a strict 800-171 mandate per the spec, but each is what a 3PAO and the customer's central security team would expect to see in a SSP. Mappings verified against [NIST SP 800-171 Rev 2](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-171r2.pdf) and [Google's NIST 800-171 compliance page](https://cloud.google.com/security/compliance/nist800-171).

- [x] **Shielded VM on every cluster VM** (vTPM + integrity monitoring; Secure Boot disabled, see caveat) — `enable_shielded_vm: true` + `shielded_instance_config: {enable_secure_boot: false, enable_vtpm: true, enable_integrity_monitoring: true}` on all 9 nodesets, login, controller in `blueprints/cluster.yaml`. Maps to §3.4.6 (least functionality), §3.14.1 (flaw remediation). **Caveat**: Secure Boot intentionally OFF because the Managed Lustre client kernel module (`lustre-client-modules-6.8.0-1047-gcp` from us-apt.pkg.dev) is not signed against Ubuntu's Secure Boot Microsoft-cert chain — `modprobe lustre` fails with "Key was rejected by service" when Secure Boot is enforced. This required a project-level override of `compute.requireShieldedVm` with `inheritFromParent: false` (the org-level enforcement requires Secure Boot, and inherited policies otherwise win). Production fix: MOK-sign the Lustre module in the Packer build (one-time work) and re-enable Secure Boot. vTPM + integrity-monitoring still cover most of the §3.4.6/§3.14.1 boot-integrity story; image build provenance + immutable Cloud Audit Logs cover the gap.
- [x] **OS Login enforced project-wide** — `gcloud compute project-info add-metadata enable-oslogin=TRUE` in `scripts/setup-project.sh` + `enable_oslogin: true` on every VM module. Project-level `compute.requireOsLogin` org policy enforcement applies. Maps to §3.5.1, §3.5.2, §3.5.3 (MFA via Google identity 2SV).
- [x] **No public IPs on any cluster VM** — `enable_public_ips: false` on every nodeset, `enable_login_public_ips: false` on login, `enable_controller_public_ips: false` on controller. Project-level `compute.vmExternalIpAccess` org policy enforcement applies. SSH still works via IAP (`gcloud compute ssh --tunnel-through-iap`). Maps to §3.13.1 (boundary protection).
- [x] **Cloud NAT in every region we deploy nodesets to** — `default-nat-router-{us-central1,us-south1,us-west1,us-east4,us-east5}` + `default-nat-{region}` with `auto-allocate-nat-external-ips` + `nat-all-subnet-ip-ranges` + ERROR-only logging. Required for outbound (`pip install torch`, GCS endpoint reach) once VMs lose public IPs. Codified in `scripts/setup-project.sh`.
- [x] **Serial port access disabled** — project-level `compute.disableSerialPortAccess` org policy enforcement applies (no override). Maps to §3.4.6, §3.4.7 (disable nonessential functions).
- [x] **Write-only audit log sink** — `my-slurm-audit-sink` in us-central1, 400-day retention, filters Cloud Audit Logs (activity / data_access / system_event) into a locked Logging bucket. Defense-in-depth on top of the immutable `_Required` sink. Maps to §3.3.8, §3.3.9.
- [ ] **VPC Service Controls perimeter** — no Cluster Toolkit support; org-level `gcloud access-context-manager perimeters create`. the customer's central security team owns this; it wraps the project (CUI enclave boundary, §3.1.3 + §3.13.1 defense-in-depth). Recommended but not strictly required.
