# Reference-Deployment Success Checklist

## Cluster deployment

- [x] **Baseline NIST 800-171 met by GCP defaults + 6 hardening upgrades applied** — see `docs/nist_800_171_hardening.md` for the full mapping (Shielded VM, OS Login, no public IPs + Cloud NAT, serial port disabled, write-only audit sink). Codified in `scripts/setup-project.sh`.
- [x] **Slurm controller in `us-central1` with cross-region nodesets on a single global VPC** — controller + login + Lustre + Filestore in us-central1-b; H200 Spot fan-out across us-south1-b, us-west1-c, us-central1-b, us-east4-b on the same `default` VPC. End-to-end verified on a cross-region us-south1-b H200 reading model + tissue from `/lustre` and writing per-step checkpoints to `/lustre`.
- [x] **Smart allocation across Calendar / Flex / Spot** — researchers `sbatch -p gpu` and Slurm picks the best available accelerator. Spot fan-out across 11 nodesets, preemption resilience via `--requeue` + Lustre checkpoints, admin-side daily Calendar/Flex polling. See `docs/capacity_strategy.md`.
- [x] **Natural requeue + cross-region failover** — same JobId moved across regions on node failure, transparent to researcher. `--requeue` + workload's `find_last_checkpoint()` resumed from the latest `/lustre` checkpoint. End-to-end verified: stopped a Spot H200 mid-run after step 1; slurm-gcp's `slurmsync.py` detected the preempt within ~30 sec, marked the node "Preempted instance", requeued the same JobId. Slurm tried us-west1-c (W2) → stockout → us-central1-b (W3) → us-south1-b reattempt (Spot recovered). Container pulled from local SSD cache (instant), gpu_run.py read `/lustre/checkpoints/$JOB/checkpoint_step1.npy` and started step 2. `SLURM_RESTART_COUNT=2` in the resumed job's env confirms multiple requeue paths fired correctly.

## Storage layer (POSIX-shaped researcher experience)

- [x] **Cross-region POSIX `/home` via Filestore + parallel `/lustre` via Managed Lustre on a single global VPC** — Filestore Basic_SSD at `/home` (NFS, in-kernel, just works on the slurm-gcp public image). Managed Lustre (18 TiB Standard tier, [locations](https://docs.cloud.google.com/managed-lustre/docs/locations)) at `/lustre` for parallel I/O / FSDP-style sharded checkpoints. Both on the project's `default` auto-mode VPC in us-central1-b; cross-region clients on the *same* VPC just work, no NCC needed ([Lustre VPC config](https://docs.cloud.google.com/managed-lustre/docs/vpc): *"The same VPC network must be specified when creating the Managed Lustre instance and client Compute Engine VMs"*). Lustre client modules baked into the cluster image via Packer (`install_managed_lustre: true` + `allow_kernel_upgrades: false` per `blueprints/build-lustre-image.yaml`) so the kernel-vs-modules version mismatch on the public image is solved at build-time. **Measured cross-region POSIX behavior under load** (us-south1-b H200 compute → us-central1-b storage):
    - **Lustre bulk read**: 11 GB model checkpoint cp `/lustre` → local `/tmp` in **25 sec single-stream** (~440 MB/s sustained)
    - **Lustre parallel checkpoint writes**: 10 × 103 MB `np.save` calls succeeded in-band with no observed retry, all visible from controller within seconds
    - **Filestore `/home` mount**: NFSv3 from `10.134.120.170` (us-central1-b) reachable cross-region without extra firewall or peering setup — only the `default-allow-internal` rule (10.128.0.0/9 → tcp/udp/icmp) was needed, codified in `setup-project.sh`
    - **GCS-FUSE `/gcs` mount**: comparable to in-region (Cloud Storage FUSE handles regional affinity automatically); stdout streamed cleanly to `gs://my-slurm-*/runs/`
    - **slurmctld ↔ slurmd heartbeat**: cross-region RTT (~32 ms us-south1↔us-central1) is dominated by the heartbeat round-trip, well under slurmd's 60s `MessageTimeout` default — zero timeouts observed across a 116-min job
- [ ] **Filestore Regional as the cost-down storage swap** — *Discovery questions: (1) Is the workload single-VM only? (2) Does `/home` throughput stay under ~25 GiB/s? (3) Are there NO parallel I/O patterns (no FSDP / ZeRO-style sharded checkpoint writes)? (4) Is the customer cost-sensitive (~$200/mo at 1 TiB minimum vs Lustre's 18 TiB minimum)?* Stays unchecked for the customer because their multi-node H200 training expectation answers "no" to (3). Cluster Toolkit makes the swap a one-line module change (`modules/file-system/filestore` with `filestore_tier: ENTERPRISE` in place of `modules/file-system/managed-lustre`).
- [x] **GCS POSIX-shaped read + write through Cloud Storage FUSE** — every cluster VM auto-mounts the project's GCS bucket at `/gcs` (Hierarchical Namespace, high-perf config auto-applied for `a3-ultragpu-8g`). Writable workspace at `/gcs/runs/<JOB>/`; read-only inputs convention at `/gcs/weights/data/`. **Discovery questions for when GCS-FUSE alone is sufficient (i.e., no Filestore/Lustre needed)**: *(1) Does the workflow have zero hard-link or file-locking dependencies? (2) Is auth NOT `.ssh`-key-based? (3) Do Python deps install per-job (not into `/home/.local`)? (4) Is the working set bulk read/append, not many-small-file stat-heavy?* For the customer's NIST 800-171 enclave with multiple researchers running multi-node training, all four answer "no" — real POSIX is needed (use Managed Lustre above).
- [ ] **Anywhere Cache + Hyperdisk ML for cross-region read acceleration & weight broadcast** — *Anywhere Cache (now Rapid Cache) eliminates the $0.02/GiB cross-region read fee on multi-region GCS buckets. Hyperdisk ML is read-only multi-attach disk for serving weights to up to 2,500 VMs in a zone.* Both stay unchecked. **Discovery questions**: *(1) Is the cross-region GCS read bill > standing Anywhere Cache cost (~hundreds of GiB/month sustained)? (2) Are 10+ VMs concurrently reading the same weights?* Until those answer yes, the customer's current 2-VM scope doesn't trigger either. Wire in when the answers flip.

## Known blockers worth remembering

- **Cluster Toolkit v1.90.0 pins `terraform == 1.12.2` exactly**, Packer is no longer in Homebrew core, default Compute SA on a fresh project has zero IAM bindings, and the `default` VPC isn't auto-created on org-policy-locked projects — all codified in `scripts/setup-project.sh`.
- **Default VPC routing-mode must be GLOBAL** for cross-region clients to reach the controller + Lustre in us-central1. Plus a `default-allow-internal` firewall rule (10.128.0.0/9 → tcp/udp/icmp) and `VmDnsSetting=GlobalOnly` project metadata (so cross-region VMs can resolve the controller's short hostname for the `/opt/apps` NFS mount). All codified in `scripts/setup-project.sh`.
- **Cluster Toolkit storage modules need explicit `outputs: - network_storage`** to flow `/home`-style mounts to a downstream group's controller. Without it, `gcluster export-outputs` skips the storage handle and the cluster group's `terraform apply` blocks on a missing input variable.
- **Logging buckets can't be created in `global` location** when `gcp.resourceLocations: us-*` org policy is enforced — `--location=us-central1` (or another allowed US region) is required. Hit this making the audit log sink bucket.
- **`compute.requireShieldedVm` org policy requires Secure Boot** (not just vTPM + integrity-monitoring) — `enforce: false` at project level only takes effect with `inheritFromParent: false`; otherwise org-level enforcement wins.
- **Secure Boot blocks the Managed Lustre kernel module** — `lustre-client-modules-*-gcp` from `us-apt.pkg.dev/projects/lustre-client-binaries` is not signed against Ubuntu's Secure Boot Microsoft-cert chain. `modprobe lustre` fails with "Key was rejected by service". Fix today: Secure Boot OFF (vTPM + integrity-monitoring still on); production fix: MOK-sign the Lustre module in the Packer build.
- **slurm-gcp public image's pre-baked Lustre modules don't match the runtime kernel** — image ships `lustre-client-modules-5.15.0-88-generic` but boots `6.8.0-1047-gcp`. Toolkit's `install-managed-lustre-client.sh` short-circuits when it sees `lustre` in `/proc/filesystems` (true on the public image because of the stale modules), so matching 6.8 modules never get installed → `mount.lustre` fails. **Fix in this repo**: `blueprints/build-lustre-image.yaml` (Packer build with `install_managed_lustre: true` + `allow_kernel_upgrades: false`) bakes the right kernel-matched modules into a custom image; `cluster.yaml` uses that image.
- **slurm-gcp v6.12.1 Ansible playbook tries to install the deprecated `stackdriver-agent`** package on Ubuntu 22.04 — the package is no longer in apt. Sidestep with `monitoring_agent: cloud-ops` in `slurm_vars.json` passed to the Packer build (the official `a3ultra-slurm-blueprint.yaml` does this; we inherit the pattern in `build-lustre-image.yaml`).
- **slurm-gcp public image's bundled docker-install task fails on H200 first boot** — `install_docker.yml exit_code=2` in `/var/log/slurm/setup.log`. We don't need docker; drop the `docker:` block from the Cluster Toolkit `a3_startup` runner.
- **Cross-region RDMA is not supported** — the GPU-RDMA VPC profile is per-zone (`<zone>-vpc-roce`). Multi-node H200 training over RoCE only works *within* the region the RDMA VPC was provisioned in (us-central1 in our blueprint). Cross-region H200 nodesets are single-node only.
- **a3-ultragpu-8g requires 0 or 8 MRDMA NICs after the first 2 GVNICs** — bulkInsert errors with "first NIC and the second NIC must be of type GVNIC, then 0 or 8 MRDMA". The single-VPC architecture only attaches 1 GVNIC (default VPC) + 8 MRDMA, which trips this validation. We dropped RDMA from `h200_spot_central1b` to fit; multi-node H200 RDMA needs a second GVNIC subnetwork added back to the primary network plus the existing RDMA secondary.
- **Cross-region `/lustre` is for bulk reads/writes only, not many-small-files** — Python imports from a `/lustre/python-deps` pip-install staging dir take 10+ minutes on a cross-region nodeset due to thousands of small `stat`/`open` syscalls × 30ms RTT. Multiplied by ~9 worker subprocesses for an 8-GPU CIFM run = 30+ min before any GPU touches the model. Container delivery (Pyxis + Enroot pulling from multi-region Artifact Registry) sidesteps this — every import is a local-SSD read after the first per-node image pull (~25-50 sec).
- **slurm-gcp's setup.py may regenerate `/usr/local/etc/slurm/plugstack.conf` at controller boot** — if so, the image-baked Pyxis plugstack entry gets clobbered. If `srun --container-image` returns "spank plugin not loaded", add a `controller_startup_script` runner in `cluster.yaml` that re-drops `plugstack.conf.d/pyxis.conf` and restarts `slurmctld` after slurm-gcp's setup completes. Untested at deploy time — will surface on first containerized SBATCH.
- **Packer image-build VM must comply with project org policies** — once `setup-project.sh` removes the project-level org-policy bypasses (Shielded VM, OS Login, no public IPs), the Packer build VM needs the matching settings or the GCE bulkInsert returns 412. Codified in `blueprints/build-lustre-image.yaml` packer module: `omit_external_ip: true`, `enable_shielded_vm: true` + `secure_boot: true`, `metadata: enable-oslogin: "TRUE"`. Note: Secure Boot is fine for the BUILD VM (it doesn't mount Lustre); cluster nodesets keep Secure Boot OFF until the Lustre kmod is MOK-signed.
- **Cloud Build API must be enabled before `gcloud builds submit`** — `setup-project.sh` enables `cloudbuild.googleapis.com` alongside `artifactregistry.googleapis.com`. Without this, container builds fail with `PERMISSION_DENIED: Cloud Build API has not been used`.
- **Stale image-build VPC state breaks `gcluster create -w` for the image blueprint** — if a prior image-build deployment dir is gone but the GCP-side VPC + NAT + firewall + addresses still exist (which is the normal state after a successful Packer build — only the image survives by design, the VPC is intentionally kept around for re-runs but the local Terraform state isn't), the next `gcluster deploy wzslurm-img` fails with 409 alreadyExists. Fix: `gcloud compute` delete the stale `wzslurm-img-*` resources in dependency order (NAT → router → addresses → firewalls → subnet → network), then redeploy. Tracked as a follow-up to make the image build fully idempotent.
